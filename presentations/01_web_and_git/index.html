Клиент-серверная архитектура

Веб работает по принципу взаимодействия двух сторон:

📱 Клиент (Client)

Тот, кто запрашивает данные. Это ваш браузер (Chrome, Safari, Firefox), мобильное приложение или консольная утилита.

🖥️ Сервер (Server)

Мощный компьютер, который постоянно включен, хранит файлы сайта, обрабатывает запросы от клиентов и отправляет им ответы.

Аналогия: Гость в ресторане (Клиент) делает заказ официанту, а кухня (Сервер) готовит блюдо и отдает его гостю.

Как они находят друг друга? IP-адрес

У каждого устройства в сети Интернет есть свой уникальный адрес.

  • IP (Internet Protocol) — уникальный числовой идентификатор устройства в сети.
  • IPv4: Состоит из 4 чисел от 0 до 255, разделенных точками (например, 192.168.1.1 или 87.250.250.242).
  • IPv6: Новый стандарт, использующий шестнадцатеричную систему записи из-за нехватки адресов IPv4 (например, 2001:0db8:85a3:0000:0000:8a2e:0370:7334).
Аналогия: IP-адрес — это почтовый адрес вашего дома. Без него почтальон (роутер) не поймет, куда нести письмо.

DNS: Телефонная книга Интернета

Людям сложно запоминать цифры вроде 77.88.55.242. Мы предпочитаем буквы: yandex.ru или google.com.

DNS (Domain Name System) — система доменных имен, переводящая понятные имена сайтов в числовые IP-адреса.

  1. Вы вводите: kvant-camp.ru
  2. Браузер спрашивает у DNS-сервера: «Какой IP у этого имени?»
  3. DNS-сервер отвечает: 185.178.104.14
  4. Браузер делает запрос на IP 185.178.104.14

Порты: Двери на сервере

На одном сервере может работать одновременно много программ: сайт, база данных, почтовый сервер.

  • Порт — числовой идентификатор конкретного процесса/программы на компьютере (от 0 до 65535).
  • Стандартные порты для веб-сайтов:
    • 80 — HTTP (простой веб, без шифрования)
    • 443 — HTTPS (защищенный веб, с шифрованием)
  • Порты для разработки (часто используются нами):
    • 8000 — стандартный порт для Django-сервера разработки
    • 5432 — СУБД PostgreSQL, 3306 — MySQL
Аналогия: IP-адрес — это адрес отеля. Порт — это номер комнаты конкретного гостя.

Знакомьтесь: Протокол HTTP

  • HTTP (Hypertext Transfer Protocol) — протокол передачи гипертекста. Это набор правил, по которым общаются клиент и сервер.
  • Работает по схеме: «Запрос (Request) → Ответ (Response)».
  • HTTP является протоколом без сохранения состояния (stateless): каждый запрос автономен и ничего не знает о предыдущих.
📱 Клиент — HTTP Запрос → 🖥️ Сервер
📱 Клиент ← HTTP Ответ —— 🖥️ Сервер

Анатомия HTTP-запроса

Запрос клиента представляет собой обычный текст определенного формата:

GET /news/ HTTP/1.1
Host: kvant-camp.ru
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64)
Accept-Language: ru-RU,ru;q=0.9

[Здесь может быть тело запроса, если отправляются данные]
  1. Стартовая строка: Метод (GET), путь к ресурсу (/news/) и версия протокола (HTTP/1.1).
  2. Заголовки (Headers): Дополнительная информация о клиенте (домен, язык, тип браузера).
  3. Тело запроса (Body): Данные, передаваемые серверу (например, пароль при авторизации).

Основные HTTP-методы: GET и POST

Характеристика GET POST
Назначение Запрос/получение данных с сервера Отправка данных на сервер (создание/изменение)
Передача параметров В адресной строке: ?search=python&page=2 В скрытом теле запроса (Body)
Безопасность данных Низкая (параметры видны в истории браузера) Выше (подходит для паролей, личных данных)
Кэширование Да, кэшируется браузером Нет, не кэшируется

Анатомия HTTP-ответа

Сервер возвращает структурированный ответ:

HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 1542
Server: gunicorn

<!DOCTYPE html>
<html><head>...</head><body>Привет, Квант!</body></html>
  1. Стартовая строка: Версия протокола и Код состояния (200 OK).
  2. Заголовки ответа: Формат данных (Content-Type), длина, серверное ПО.
  3. Тело ответа: Запрошенный контент (HTML-страница, картинка, JSON-данные).

Коды состояния (Status Codes)

Трехзначное число, сообщающее о результате обработки запроса:

  • 2xx — Успех (Success):
    • 200 OK — все прошло успешно.
    • 201 Created — успешно создано.
  • 3xx — Перенаправление (Redirect):
    • 301/302 — перейдите по другому адресу.
  • 4xx — Ошибка клиента (Client Error):
    • 400 Bad Request — неверный запрос.
    • 403 Forbidden — нет доступа.
    • 404 Not Found — страница не существует.
  • 5xx — Ошибка сервера (Server Error):
    • 500 Internal Server Error — сломался код на сервере.

Что такое Git?

Git — распределенная система контроля версий (VCS). Она позволяет:

  • Сохранять историю изменений проекта (кто, что и когда изменил).
  • Откатываться к любой предыдущей версии кода.
  • Разрабатывать проект параллельно в командах, не ломая код друг друга.

Забудьте про:

project_final.zip

project_final_v2_work.py

Git заменяет этот хаос одной чистой историей коммитов.

Как работает Git: Три области

Файлы в Git проходят три ключевых этапа перед сохранением в истории:

1. Working Directory

Рабочая папка с файлами. Изменения еще не отслеживаются.

2. Staging Area (Index)

Черновик коммита. Сюда мы временно откладываем файлы для сохранения.

git add →
3. Git Repository (.git)

База данных проекта. Здесь хранятся завершенные снимки (коммиты).

git commit →

Первоначальная настройка Git

Перед началом работы нужно представиться Git, чтобы авторы коммитов были видны корректно:

# Задаем имя пользователя (латиницей)
git config --global user.name "Ivan Ivanov"

# Задаем email пользователя
git config --global user.email "ivan.ivanov@yandex.ru"

# Проверяем настройки
git config --list
Важно: Делайте это один раз на своем компьютере. Имя и email будут привязаны к каждому вашему коммиту.

Создание репозитория

Чтобы Git начал отслеживать изменения в папке вашего проекта:

# Переходим в папку проекта (в терминале)
cd my_project

# Инициализируем репозиторий
git init

# Проверяем текущее состояние репозитория
git status

Команда git init создает скрытую папку .git. В ней Git хранит всю историю изменений проекта.

Добавление изменений в индекс

Для добавления файлов в черновик коммита (Staging Area) используется команда git add:

# Добавить конкретный файл в индекс
git add main.py

# Добавить все измененные файлы в текущей папке
git add .

# Проверить статус добавленных файлов
git status
Файлы перейдут из статуса Untracked (неотслеживаемые) или Modified в статус Changes to be committed (готовые к коммиту).

Фиксация изменений: Коммит

Коммит (Commit) — это моментальный снимок (сохранение) ваших файлов. Создается командой:

git commit -m "Добавить начальную структуру сайта и файл main.py"

Правила хороших коммитов:

  • Пишите понятные сообщения на одном языке (русский или английский).
  • Сообщение должно отвечать на вопрос: «Что делает этот коммит?»
  • Не делайте огромные коммиты раз в неделю. Лучше делать маленькие логические коммиты часто.
Аналогия: Коммит — это точка быстрого сохранения (Quick Save) в компьютерной игре перед сложным уровнем.

Просмотр истории: git log

Чтобы увидеть историю всех коммитов в репозитории:

# Показать полную историю
git log

# Показать краткую историю (одна строка на коммит)
git log --oneline

Вывод git log содержит:

  • Уникальный хеш коммита (например, a1b2c3d...) — его идентификатор.
  • Имя автора и дату создания.
  • Сообщение коммита.

Git vs GitHub и удаленная работа

  • Git — инструмент на вашем локальном компьютере.
  • GitHub / GitLab — облачные сервисы для хранения ваших Git-репозиториев и совместной работы.

Базовые команды для работы с удаленным репозиторием:

# Клонировать существующий проект с GitHub к себе
git clone https://github.com/user/project.git

# Привязать локальный репозиторий к удаленному
git remote add origin https://github.com/user/project.git

# Отправить свои локальные коммиты в облако
git push -u origin main

# Забрать свежие изменения других разработчиков из облака
git pull
