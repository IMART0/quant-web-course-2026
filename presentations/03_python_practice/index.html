JSON-расписание: Сериализация

Сайт лагеря получает расписание смены от внешнего API в формате JSON (строка). Для работы в Python мы парсим его в словарь или список:

  • json.loads(): строка JSON ➡️ структуры Python (dict, list).
  • json.dumps(): структуры Python ➡️ строка JSON.
Осторожно: опечатка в JSON (например, одинарные кавычки или True вместо true) вызовет json.JSONDecodeError. Используйте try-except! 
import json

# JSON-строка (ключи строго в " ")
raw_schedule = '[{"title": "OpenTalk", "active": true}]'

try:
    schedule = json.loads(raw_schedule)
    print(schedule[0]["title"]) # OpenTalk
except json.JSONDecodeError as e:
    print(f"Ошибка в формате JSON: {e}")

Лента новостей: Работа с datetime

Новости на сайте лагеря публикуются в разное время. Чтобы отсортировать их, нужно перевести строки в объекты datetime:

  • strptime(str, format) — парсинг строки в объект по шаблону.
  • strftime(format) — красивое форматирование даты для вывода на сайт.
  • timedelta — интервалы времени (например, длительность мастер-класса).
from datetime import datetime, timedelta

# Шаблон: День.Месяц.Год Часы:Минуты
fmt = "%d.%m.%Y %H:%M"
pub_time = datetime.strptime("26.06.2026 14:30", fmt)

# Прибавим время (длительность лекции)
lecture_end = pub_time + timedelta(minutes=90)
print(lecture_end.strftime("%H:%M")) # 16:00

Спикеры и лекции: Поиск и фильтрация

На сайте «Кванта» пользователи хотят искать мероприятия конкретного лектора.

  • Поиск должен быть регистронезависимым (приводим к .lower()).
  • Поиск должен поддерживать частичные совпадения (оператор in).
  • Обеспечиваем защиту от отсутствующих полей (метод .get()).
events = [
    {"title": "Веб на Django", "speaker": "Игорь"},
    {"title": "Физика космоса", "speaker": "Сергей"}
]

# Фильтрация
query = "иг"
result = [e for e in events if query.lower() in e.get("speaker", "").lower()]
# Результат: [{"title": "Веб на Django", "speaker": "Игорь"}]

Хранилище участников: Mock БД

Сайту нужна база данных для хранения участников смены. В учебных целях мы напишем In-Memory Database на основе вложенных словарей Python:

self._data = {

Важнейшее правило ОЗУ-БД

Словари в Python передаются по ссылке. Если мы не сделаем глубокую копию с помощью copy.deepcopy(), внешние изменения изменят данные в самой базе данных!

URL-Router: Связываем пути с кодом

Роутер анализирует URL запроса и находит соответствующий обработчик (View):

  • Простые пути: /news/ ➡️ show_news().
  • Сложнее — пути с параметрами: /speaker/<string:username>/ ➡️ speaker_profile(username).
Мы напишем парсер путей, который разбивает URL на сегменты (по слэшу) и динамически вырезает переменные из строки запроса. 
# Как это будет выглядеть:
@router.register("/event/<int:event_id>/", "GET")
def get_event(event_id):
    return f"Мероприятие №{event_id}"

# При вызове /event/15/
# Роутер вернет функцию get_event и {"event_id": 15}

Запрос проходит цепочку посредников (Middleware), маршрутизируется к обработчику, который запрашивает данные из БД и возвращает ответ.

Middleware: Цепочка обработки

Middleware — слои, через которые проходит запрос до и после попадания во View.

  • Логирование: записывает в лог информацию о запросе.
  • Авторизация: проверяет токен. Если токена нет, возвращает 401 Unauthorized, прерывая цепочку.

Принцип конвейера

Каждое Middleware получает функцию next_handler. Если оно не вызывает next_handler(), обработка прерывается. Это защищает чувствительные разделы сайта.
