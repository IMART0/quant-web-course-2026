Напоминание: Архитектура MVT

Django работает по паттерну MVT (Model-View-Template):

  • Model (Модель): Описывает структуру данных и взаимодействует с БД.
  • Template (Шаблон): HTML-страница, определяющая внешний вид.
  • View (Представление): Логическое ядро. Принимает запросы, запрашивает данные у моделей и отправляет их в шаблоны.

Сегодня мы сфокусируемся на Views (Представлениях) и Маршрутизации (Routing).

Как это работает?

Пользователь переходит по URL → Django находит нужный View → View обрабатывает запрос → View возвращает ответ (HTML-текст, JSON или ошибку).

Django-проект vs Django-приложение

В Django строго разделяются понятия Project (Проект) и Application (Приложение):

Project (Проект)

Это весь сайт в целом. Он содержит глобальные настройки сайта, параметры баз данных и корневые маршруты (файлы settings.py, urls.py проекта).

Application (Приложение)

Это независимый модуль, решающий конкретную задачу. Например, на сайте ЛФМШ «Квант» приложениями будут: camp (информация, расписание, отряды), study (курсы и задачи), blog (новости смены).

Один Django-проект объединяет множество независимых приложений!

Создание приложения (camp)

Для создания нового приложения используется встроенная утилита управления:

python manage.py startapp camp

Эта команда создаст папку camp со стартовой структурой файлов внутри нашего проекта.

Правило именования: Название пишется строчными латинскими буквами, обычно в единственном или множественном числе (например, camp, users, events), без спецсимволов.

Анатомия Django-приложения

Внутри созданной папки camp/ Django сгенерировал базовые файлы:

  • migrations/ — папка для отслеживания изменений в базе данных.
  • admin.py — регистрация моделей приложения в админ-панели.
  • apps.py — технический файл конфигурации приложения.
  • models.py — описание структур данных (Модели).
  • tests.py — тесты для проверки работоспособности кода.
  • views.py — представления (наша бизнес-логика).
  • urls.py (создаем вручную) — файл локальных маршрутов приложения.

Регистрация приложения в settings.py

Чтобы Django знал о нашем приложении, его необходимо добавить в список INSTALLED_APPS файла settings.py проекта:

# mysite/settings.py

INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    # ... другие системные приложения ...
    'camp.apps.CampConfig', # Наш новый класс конфигурации
]
Внимание: Без регистрации приложения в INSTALLED_APPS Django не сможет найти его шаблоны, модели и локальные команды!

Что такое View (Представление)?

View — это функция или класс на Python, которая принимает HTTP-запрос и возвращает HTTP-ответ.

Основной цикл Django View:

HttpRequest → View (Логика обработки) → HttpResponse

Представление обязано возвращать корректный объект ответа (например, HttpResponse с текстом, редирект или ошибку). Если функция ничего не вернет, Django вызовет ошибку.

Написание первой View (FBV)

Создадим простое представление на основе функций (Function-Based View):

# camp/views.py
from django.http import HttpResponse

def main_page(request):
    return HttpResponse("Добро пожаловать на официальный сайт ЛФМШ «Квант»!")
  • request — объект класса HttpRequest. Содержит все метаданные запроса (GET/POST параметры, IP-адрес, тип браузера).
  • HttpResponse — упаковывает строковый ответ в корректный HTTP-пакет с кодом состояния 200 OK.

Маршрутизация (Routing) в Django

Маршрутизация связывает URL-адрес из строки браузера с конкретной View. Она описывается в файле urls.py.

  • URL Dispatcher — перебирает список маршрутов сверху вниз до первого совпадения.
  • Для создания маршрутов используется функция path().
# Синтаксис функции path
path(route, view, name=None)
  • route: шаблон пути (строка, например 'schedule/').
  • view: ссылка на функцию (например, views.schedule_list).
  • name: имя маршрута (для обратного поиска).

Делегирование маршрутов (include)

Не пишите все маршруты в глобальном urls.py проекта! Используйте функцию include() для подключения локальных маршрутов приложений:

# mysite/urls.py (Проект)
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('', include('camp.urls')), # Делегируем корень сайта приложению camp
]
Благодаря include() приложение camp становится изолированным. Мы можем перенести его в другой проект, и его внутренние ссылки не сломаются.

Локальные маршруты приложения (camp/urls.py)

Создадим файл urls.py в папке приложения camp/:

# camp/urls.py (Приложение)
from django.urls import path
from . import views

urlpatterns = [
    path('', views.main_page, name='main_page'),
    path('schedule/', views.schedule_list, name='schedule_list'),
]
Так как в проекте мы подключили маршруты через path('', include('camp.urls')), пустая строка '' соответствует корню сайта (/), а 'schedule/' — адресу /schedule/.

Схема обработки запроса в Django

Пользователь запрашивает URL: /schedule/
  ↓
Django парсит корневой mysite/urls.py
  ↓ находит совпадение с '' → передает в include('camp.urls')
Django парсит локальный camp/urls.py
  ↓ находит совпадение с 'schedule/' → вызывает views.schedule_list(request)
Функция views.schedule_list возвращает HttpResponse с расписанием
  ↓
Пользователь видит страницу в браузере (статус 200 OK)

Динамическая маршрутизация

В лагере каждый день проводятся десятки мероприятий. Как открыть детали конкретного события по его ID без создания сотен статичных маршрутов?

Используйте угловые скобки для захвата переменных из URL:

# camp/urls.py
urlpatterns = [
    # ...
    path('schedule/<int:event_id>/', views.event_detail, name='event_detail'),
]
  • int — конвертер пути (гарантирует, что параметр является целым числом).
  • event_id — имя переменной, которая передается в аргументы представления.

Встроенные конвертеры путей

Конвертер Описание Пример соответствия
<int:...> Целые неотрицательные числа (1, 2, 105). /schedule/5/
<str:...> Любая непустая строка, кроме слеша / (по умолчанию). /teacher/ivanov/
<slug:...> Буквы, цифры, дефисы и символы подчеркивания. /teacher/alex-smirnov/
<uuid:...> Стандартизированный идентификатор UUID. /event/9f7e8a93-b.../
<path:...> Любая непустая строка, включая слеши /. /files/physics/lec1.pdf

Обработка динамических параметров во View

Имя переменной в urls.py должно символ в символ совпадать с именем аргумента во View:

# camp/views.py

# Соответствует маршруту: path('schedule/<int:event_id>/', views.event_detail)
def event_detail(request, event_id):
    # event_id уже имеет тип int благодаря конвертеру
    return HttpResponse(f"Детальная информация о мероприятии №{event_id}")
Django автоматически преобразует строку из URL в число int перед передачей во View. Вам не нужно писать int(event_id) вручную!

Управление HTTP-статусами ответов

Мы можем возвращать кастомные статус-коды (например, 404 Not Found, 400 Bad Request) с помощью параметра status или специализированных классов:

# Способ 1: Использование именованного параметра status
def bad_request_demo(request):
    return HttpResponse("Некорректный запрос", status=400)

# Способ 2: Использование готовых наследников HttpResponse
from django.http import HttpResponseNotFound

def teacher_profile(request, username):
    # Если имя преподавателя пустое или не найдено
    if username == "admin":
        return HttpResponseNotFound("Профиль администратора недоступен")
    return HttpResponse(f"Профиль преподавателя: {username}")

Генерация ошибки 404 с помощью исключений

В процессе работы логики (например, если ID события не найден) бывает удобнее прервать выполнение функции и выбросить исключение Http404:

from django.http import Http404

EVENTS = {1: "Лекция по Django", 2: "Физический квест"}

def event_detail(request, event_id):
    if event_id not in EVENTS:
        raise Http404("Мероприятие не найдено в расписании!")
    return HttpResponse(f"Мероприятие: {EVENTS[event_id]}")
Помни: Исключение Http404 выбрасывается с помощью raise, а не возвращается с помощью return!

Частые ошибки при написании View и URL

  • Забытый return:
    Ошибка: "The view didn't return an HttpResponse object.". View обязана возвращать ответ!
  • Несовпадение имен аргументов:
    В URL: <int:event_id>, во View: def detail(request, ev_id). Возникнет TypeError.
  • Отсутствие слеша в конце:
    По умолчанию пишите слеш в путях: path('schedule/', ...), иначе Django не сможет сопоставить адрес.
  • Забытая запятая:
    Списки urlpatterns в Python требуют запятую после каждого элемента.

Рекомендации по разработке (Best Practices)

  • Именуйте маршруты:
    Всегда добавляйте name='...' в path() для гибкого управления ссылками.
  • Изолируйте URL:
    Корневой urls.py должен быть точкой входа, подключающей файлы приложений через include().
  • Концепция «Тонких контроллеров»:
    Views должны отвечать только за маршрутизацию и выдачу ответов. Расчеты и выборку данных лучше делать в моделях.
  • Типизируйте параметры:
    Всегда вешайте строгие конвертеры (например, <int:id> вместо <str:id>) для защиты от мусорного трафика.

Резюме и практические задания

  • startapp: создает изолированную логическую область проекта.
  • INSTALLED_APPS: регистрирует новые приложения.
  • View: связующее звено, генерирующее HttpResponse из HttpRequest.
  • include: делегирует парсинг путей локальному urls.py.
  • Конвертеры путей: валидируют URL-параметры на лету.
  • Http404 и статус-коды: информируют о корректности выполнения запроса.
