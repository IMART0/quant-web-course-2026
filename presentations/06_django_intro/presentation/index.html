Библиотека vs Фреймворк

Библиотека (Library)

Вы контролируете процесс. Вы сами вызываете функции библиотеки тогда, когда вам это нужно.

Пример: math.sqrt(), requests.get(). Разработчик — главный.

Фреймворк (Framework)

Фреймворк контролирует вас. Вы вставляете свой код в готовые "слоты", а фреймворк сам его запускает.

Инверсия управления (IoC): Принцип Голливуда — «Не звоните нам, мы сами вам позвоним».

Философия Django: «Батарейки в комплекте»

  • Batteries Included: В комплекте уже идут СУБД (SQLite), админ-панель, система авторизации, защита от CSRF/SQL-инъекций и генератор HTML-шаблонов.
  • DRY (Don't Repeat Yourself): Не повторяйся. Фреймворк минимизирует дублирование кода.
  • Создан для скорости: Родился в редакции новостной газеты в 2003 году для быстрой публикации материалов журналистами. Назван в честь джазового гитариста Джанго Рейнхардта.
Результат для нас: Мы сосредоточимся на логике сайта «Квант», а не на написании систем авторизации с нуля.

Классический паттерн MVC

  • Model (Модель): Работа с данными и базой данных. Хранит структуру и правила валидации.
  • View (Представление): Внешний вид приложения. HTML-страница, которую видит пользователь.
  • Controller (Контроллер): "Мозг". Принимает запросы от пользователя, просит данные у Модели и передает их в Представление.

Аналогия с рестораном:

🍳 Кухня и склад = Модель

📋 Меню и оформленное блюдо = Представление (View)

🤵 Официант = Контроллер

Архитектура MTV в Django

В Django используется адаптированная версия паттерна MVC, которая называется MTV (Model-Template-View):

  • M — Model (Модель): База данных и её описание в виде Python-классов (ORM).
  • T — Template (Шаблон): HTML-страница со специальным синтаксисом (шаблонизатором) для вывода динамических данных. (В MVC это View).
  • V — View (Представление): Python-функция или класс, содержащий логику обработки запроса. (В MVC это Controller).
⚠️ Внимание: Не путайте! View в Django — это логика (контроллер), а не визуальное оформление!

Соответствие паттернов

Классический MVC Django MTV Роль в проекте Где лежит в коде
Model Model Описание таблиц БД и данных models.py
View Template HTML-верстка, стили, разметка templates/*.html
Controller View Бизнес-логика, обработка HTTP-запросов views.py
— URL Conf Диспетчер запросов (распределяет по View) urls.py

Подготовка окружения

Каждому Python-проекту требуется изолированное виртуальное окружение, чтобы зависимости не конфликтовали.

# 1. Создаем виртуальное окружение venv
python -m venv venv

# 2. Активируем его (в PowerShell / терминале Windows)
venv\Scripts\activate

# 3. Устанавливаем Django 5+
pip install django
💡 После активации окружения в начале строки терминала появится префикс (venv).

Инициализация проекта kvant_site

Создаем наш проект под именем kvant_site. Обязательно используйте точку на конце!

django-admin startproject kvant_site .

С точкой (ПРАВИЛЬНО):

Управляющий файл manage.py создается прямо в текущей папке, а конфигурация — внутри kvant_site/. Нет лишней вложенности.

Без точки (НЕПРАВИЛЬНО):

Django создаст внешнюю папку kvant_site/, а в ней внутреннюю kvant_site/ и manage.py. Получится неудобная "матрешка".

Анатомия проекта kvant_site

  • 📂 kvant_site/ — конфигурационный пакет нашего проекта.
    • 📄 __init__.py — маркер Python-пакета.
    • 📄 settings.py — главный конфигурационный файл проекта.
    • 📄 urls.py — диспетчер URL-адресов (маршруты сайта).
    • 📄 wsgi.py / asgi.py — точки входа для боевых веб-серверов.
  • 📄 manage.py — скрипт управления проектом через терминал. Код в нем не редактируем!

Проект vs Приложение (Project vs App)

Проект (Project)

Это весь веб-сайт целиком. Он объединяет в себе настройки, базы данных, маршруты и отдельные функциональные модули (приложения).

Приложение (App)

Это независимый, изолированный модуль внутри проекта, решающий одну конкретную задачу (например, форум, блог, расписание).

🔥 Одно приложение можно легко перенести из одного проекта Django в другой!

Создание приложений для kvant_site

Для нашего сайта лагеря мы выделим три отдельных модуля:

  1. main: Главная страница сайта, новости, контакты.
  2. users: Профили школьников и вожатых, авторизация, личные кабинеты.
  3. schedule: Сетка занятий, физико-математические курсы, лекции и события.
python manage.py startapp main
python manage.py startapp users
python manage.py startapp schedule

Регистрация приложений в settings.py

После создания приложений их нужно "прописать" в главном конфигурационном файле kvant_site/settings.py:

# kvant_site/settings.py
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    ...
    # Наши приложения для сайта школы Квант:
    'main.apps.MainConfig',
    'users.apps.UsersConfig',
    'schedule.apps.ScheduleConfig',
]
⚠️ Если забыть прописать приложение в INSTALLED_APPS, Django не увидит его модели базы данных и шаблоны!

settings.py: BASE_DIR и SECRET_KEY

  • BASE_DIR: Корневой путь к проекту. Вычисляется автоматически через библиотеку pathlib: 
    BASE_DIR = Path(__file__).resolve().parent.parent
    Это позволяет запускать проект на любом компьютере и сервере без жесткой привязки к путям.
  • SECRET_KEY: Уникальный криптографический ключ. Используется для подписи cookies, сессий и токенов безопасности.
    🤫 Никогда не коммитьте боевой SECRET_KEY на GitHub! Если он утек, его нужно немедленно сменить.

settings.py: DEBUG и ALLOWED_HOSTS

DEBUG = True

  • Режим разработки: При возникновении ошибок Django выводит подробный traceback прямо в браузер.
  • Автоматически включен автоперезапуск сервера (Auto-reload).

DEBUG = False

  • Боевой режим (Production): Вместо отладки — лаконичные страницы 404 и 500.
  • Обязательно заполнение ALLOWED_HOSTS (список разрешенных доменов сайта).
❌ Оставлять DEBUG = True на реальном сервере в сети — грубейшая ошибка безопасности!

Настройки локализации и СУБД

Сделаем так, чтобы админка заговорила по-русски, а время соответствовало часовому поясу лагеря:

LANGUAGE_CODE = 'ru-ru'
TIME_ZONE = 'Europe/Moscow' # московское время
USE_I18N = True
USE_TZ = True

База данных по умолчанию: SQLite. Хранится в одном файле db.sqlite3 в корне.

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': BASE_DIR / 'db.sqlite3',
    }
}

Запуск локального сервера

Запуск осуществляется через терминал с помощью manage.py:

python manage.py runserver
  • Хост по умолчанию: http://127.0.0.1:8000/ (или localhost:8000). Это адрес вашего локального ПК.
  • Auto-reload: Сервер отслеживает изменения в файлах .py и автоматически перезапускается при их сохранении.
🎉 При успешном запуске перейдите по ссылке в браузере — вы увидите стартовую ракету Django!

Полезные команды manage.py

  • check: Статическая проверка конфигурации проекта на опечатки и ошибки. 
    python manage.py check
  • migrate: Применяет начальные миграции базы данных (создает таблицы для админки и пользователей). 
    python manage.py migrate
  • shell: Интерактивная консоль Python, предварительно загрузившая все настройки Django. 
    python manage.py shell
