Изучение функции render(), структуры папок и словаря context

Проблема: HTML внутри Python-кода Views

До сих пор мы формировали ответ в виде текстовых строк прямо во View:

from django.http import HttpResponse

def news_view(request):
    title = "Турнир по робототехнике"
    # ❌ Смешивание логики Python и HTML-верстки:
    html = f"<html><body><h1>Новость: {title}</h1></body></html>"
    return HttpResponse(html)

Минусы смешанного подхода:

  • Неудобство: Трудности написания CSS, JS и больших HTML-страниц.
  • Хрупкость: Любая опечатка ломает Python-код бэкенда.
  • Коллаборация: Дизайнеры не могут править верстку.
⚠️

«Спагетти-код»

Нарушается базовый паттерн разработки: смешивание логики приложения с его визуальным представлением.

Решение: Паттерн MVT в Django

Django разделяет обязанности компонентов по схеме Model-View-Template:

View (Представление)

Обрабатывает HTTP-запросы, извлекает данные (списки новостей, преподавателей, событий), упаковывает их в словарь context и вызывает шаблон.

Template (Шаблон)

HTML-файл с разметкой DTL (Django Template Language). Служит каркасом, определяющим, как отобразить переданные данные.

Принцип разделения ответственности: View отвечает за что отобразить (данные), а Template — за как отобразить (дизайн страницы).

Где Django ищет шаблоны?

Существует два основных подхода к хранению HTML-файлов:

1. Глобальная папка (Project-level)

Общие шаблоны для всего сайта (например, базовая верстка, футер, шапка).

kvant_site/
├── templates/      # Глобальная папка
│   └── base.html
└── settings.py

2. Папка приложения (App-level)

Изолированные шаблоны конкретных модулей (новости, расписание, контакты).

kvant_site/
└── news/
    └── templates/   # Папка внутри app
        └── news/
            └── list.html
Важное правило (Пространство имен): Шаблоны внутри приложения кладут в подпапку с тем же именем: templates/news/list.html. Это предотвращает коллизии имен!

Конфигурация TEMPLATES в settings.py

За поиск шаблонов отвечает параметр TEMPLATES в файле настроек:

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [BASE_DIR / 'templates'], # Указываем путь к глобальным шаблонам
        'APP_DIRS': True,                 # Автопоиск в папках templates/ приложений
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                ...
            ],
        },
    },
]
  • DIRS: Списки абсолютных путей, где Django будет искать шаблоны в первую очередь.
  • APP_DIRS: True: Фреймворк ищет шаблоны внутри каждого приложения из INSTALLED_APPS.

Функция render()

Для загрузки HTML-шаблона и его заполнения данными используется функция-хелпер render():

from django.shortcuts import render

def index_view(request):
    # Логика сбора данных
    return render(request, 'news/list.html', context={})

Параметры функции render:

  1. request — исходный объект HTTP-запроса (обязательный аргумент).
  2. template_name — относительный путь к файлу шаблона (например, 'news/list.html').
  3. context — необязательный словарь с динамическими данными (по умолчанию пуст).

Результат работы: Возвращает объект HttpResponse с готовым HTML-кодом.

Контекст шаблона (Context Dictionary)

Контекст — это Python-словарь, ключи которого становятся именами переменных в HTML-шаблоне:

# В файле views.py
context = {
    'camp_name': 'Квант',
    'shift_year': 2026,
    'news_list': [
        'Приезд лекторов',
        'Турнир по футболу'
    ]
}

Что можно передавать?

  • Простые типы: строки, числа, логические флаги.
  • Коллекции: списки, словари, кортежи.
  • Сложные объекты: объекты пользовательских классов, инстансы моделей.
  • Методы объектов (вызываются автоматически).

Язык шаблонов Django (DTL)

DTL содержит 4 основных синтаксических блока для управления разметкой:

Синтаксис Тип Назначение Пример
{{ variable }} Переменные Вывод значений в HTML {{ camp_name }}
{% tag %} Теги Управляющие конструкции {% for n in news_list %}
{{ var|filter }} Фильтры Форматирование вывода {{ camp_name|upper }}
{# comment #} Комментарии Бэкенд-комментарии {# Скрыто от клиента #}

Вывод переменных и обращение через точку

Для доступа к вложенным структурам в DTL используется единый оператор — точка (.):

# views.py
teacher = {
    'name': 'Иван Иванов',
    'subject': 'Математика',
    'contacts': {'phone': '123-45'}
}

Обращение к элементам:

  • Ключ словаря: {{ teacher.name }}
  • Вложенный ключ: {{ teacher.contacts.phone }}
# views.py
news = ['Новость 1', 'Новость 2']
# Наследует методы строк и списков
  • Индекс списка: {{ news.0 }} (первый элемент)
  • Метод (без скобок): {{ teacher.name.upper }}
Важно: Вызов методов со скобками {{ name.upper() }} в DTL вызовет синтаксическую ошибку. Параметры передавать нельзя!

Фильтры шаблонов: Изменение вывода данных

Фильтры применяются с помощью вертикальной черты | и позволяют преобразовывать данные прямо в шаблоне:

  • {{ camp_name|upper }} → перевод в верхний регистр (КВАНТ).
  • {{ teacher.bio|default:"Биография отсутствует" }} → значение по умолчанию.
  • {{ news_list|length }} → количество элементов в списке.
  • {{ event.date|date:"d.m.Y" }} → форматирование даты.
  • {{ event.title|truncatewords:5 }} → обрезка длинного текста.
  • {{ teacher.rating|floatformat:1 }} → округление вещественного числа.
Цепочки фильтров: Фильтры можно комбинировать по цепочке: {{ event.text|default:"нет текста"|truncatewords:10 }}. Обработка идет слева направо.

Безопасность: Автоэкранирование и фильтр safe

По умолчанию Django автоматически экранирует спецсимволы HTML (<, >, &, ", '). Это защищает приложение от XSS-атак (внедрения вредоносных скриптов).

Пример из views.py:

context = {
    'tag_html': '<b>Срочно!</b>'
}
  • Обычный вывод {{ tag_html }} отобразится в браузере как текст: <b>Срочно!</b>.
  • Браузер не интерпретирует HTML.

Отключение защиты через фильтр safe:

# В шаблоне HTML:
{{ tag_html|safe }}
  • Результат в браузере: Срочно! (жирный шрифт).
  • Используйте |safe только для проверенных данных, сгенерированных вами.

Вывод списков: Тег {% for %}

Для отображения динамических списков (новостей, событий, преподавателей) используется тег for:

<ul>
    {% for teacher in teachers %}
        <li>{{ forloop.counter }}. {{ teacher.name }} — {{ teacher.subject }}</li>
    {% empty %}
        <li>Преподаватели еще не добавлены в список.</li>
    {% endfor %}
</ul>
  • {% empty %}: Опциональный блок. Отрендерится, если переданный список пуст или равен None.
  • forloop.counter: Служебная переменная, хранящая текущий шаг цикла (начиная с 1).
  • forloop.first / forloop.last: Булевы флаги для первой и последней итерации.

Ветвление в шаблоне: Тег {% if %}

Используется для управления показом блоков верстки в зависимости от условий:

{% if event.capacity <= event.current_registrations %}
    <span class="badge badge-accent">Мест нет</span>
{% elif event.current_registrations >= 15 %}
    <span class="badge badge-secondary">Осталось мало мест</span>
{% else %}
    <span class="badge badge-primary">Регистрация открыта</span>
{% endif %}

Возможности условных выражений в DTL:

  • Логические операторы: and, or, not.
  • Операции сравнения: ==, !=, <, >, <=, >=.
  • Проверка вхождения: in, not in (например, {% if user in event.registrations %}).

Комментарии: DTL vs HTML

Однострочные комментарии DTL

{# Это комментарий DTL. Он не попадет в HTML #}

Многострочные комментарии DTL

{% comment "Пояснение разработчику" %}
  <p>Этот кусок кода полностью</p>
  <p>будет проигнорирован Django</p>
{% endcomment %}

HTML-комментарии (Опасность!)

<!-- <h1>{{ user.password }}</h1> -->
Будьте осторожны: HTML-комментарии <!-- --> скрывают элементы на экране, но отправляются в браузер. Тэги DTL внутри них продолжают выполняться на сервере, что может привести к утечке конфиденциальных данных!

Архитектурные правила: Разделение логики

Что ДЕЛАТЬ в шаблоне (Можно)

  • Форматировать данные (выровнять регистр, дату, округлить число).
  • Перебирать коллекции для генерации списков/таблиц.
  • Переключать видимость блоков с помощью простых ветвлений.

Чего НЕ ДЕЛАТЬ в шаблоне (Нельзя)

  • Делать SQL-запросы в БД.
  • Вызывать сложные расчеты, тяжелые алгоритмы.
  • Изменять состояние объектов (сохранять данные).
  • Записывать пароли или токены.

Помните: шаблон должен быть глупым и заниматься исключительно выводом.

Типичные ошибки при работе с шаблонами

  • Ошибка TemplateDoesNotExist:
    • Забыли зарегистрировать приложение в INSTALLED_APPS.
    • Неверно написан путь в DIRS в settings.py или опечатка в имени файла.
  • Опечатки в названиях переменных:
    • Если во View передан 'teachers', а в шаблоне написано {{ teacher_list }}, Django промолчит и выведет пустую строку. Будьте внимательны!
  • Вызов методов со скобками:
    • Попытка написать {{ name.lower() }} приведет к игнорированию или ошибке разбора шаблона.
