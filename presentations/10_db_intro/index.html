Проблема: Почему не хранить данные в файлах?

Представьте, что мы пишем сайт школы «Квант» и храним баланс коинов и расписание в файле data.json.

  • Нет конкурентного доступа: Если 5 вожатых одновременно начислят баллы ребятам, файл перезапишется с ошибкой, и часть данных пропадет.
  • Медленный поиск: Чтобы найти одного студента из 10 000, серверу придется прочитать весь файл с начала.
  • Нет контроля связности: Легко случайно привязать достижение к несуществующему ID студента.
Решение: Использование специализированного ПО — СУБД, которая берет на себя управление безопасностью, скоростью поиска и целостностью данных.

Базы данных и СУБД

База данных (БД) — это упорядоченное структурированное хранилище информации.

СУБД (Система управления базами данных) — программное обеспечение для создания баз данных, чтения, записи и обновления информации.

SQLite (Встроенная СУБД)

Хранит всю БД в одном файле на диске. Не требует настройки серверов. Идеальна для обучения и небольших приложений.

PostgreSQL / MySQL (Серверные СУБД)

Работают как отдельные мощные службы. Обеспечивают одновременную работу миллионов пользователей.

Реляционная модель: Таблицы

Слово relation означает «отношение». В базах данных отношение представляется в виде двумерной таблицы.

id [PK] username first_name role
1 ivan_q Иван student
2 masha_math Мария student
3 anton_py Антон teacher
  • Столбец (Поле) — атрибут сущности. Имеет строгий тип данных (число, строка, дата).
  • Строка (Запись) — описание одного конкретного объекта.
  • СУБД контролирует типы полей и не позволит записать текст в числовое поле.

Первичный ключ (Primary Key)

Первичный ключ (PK) — уникальный идентификатор строки в таблице. СУБД использует его для поиска конкретной записи и связывания таблиц.

Свойства Первичного Ключа (PK):
  1. Уникальность: В таблице не может быть двух строк с одинаковым значением PK.
  2. Непустота: PK не может принимать значение NULL (пустота).
  3. Стабильность: PK не должен меняться (поэтому естественные ключи, такие как телефон или email, заменяют суррогатным автоинкрементным id).

Схема базы данных сайта «Квант»

Для сайта нашей школы мы проектируем структуру из связанных таблиц:

  • users: все пользователи (id, username, first_name, last_name, role).
  • students: профили учеников (user_id [PK], cohort_number, balance).
  • teachers: профили преподавателей (user_id [PK], subject, bio).
  • events: мероприятия лагеря (id, title, description, points, event_date).
  • achievements: грамоты студентов (id, student_id [FK], title, points, awarded_at).

Внешний ключ (Foreign Key)

Внешний ключ (FK) — колонка в таблице, содержащая значение первичного ключа (PK) другой таблицы. Это инструмент связывания таблиц.

Родительская таблица (students)

user_id [PK] balance
1 150
2 300

Дочерняя таблица (achievements)

id [PK] student_id [FK] title
10 1 (Иван) Турнир по шахматам
11 2 (Мария) Проект на Django

Типы связей: Один к одному (1:1)

Каждой строке первой таблицы соответствует ровно одна строка второй таблицы.

Пример в ЛШ «Квант»: Связь между пользователем users и его ролью студента students.
  • Каждая запись в students имеет user_id, который одновременно является её Primary Key и ссылается на Primary Key в users.
  • На эту связь накладывается ограничение уникальности (UNIQUE), чтобы у одного пользователя не могло появиться двух профилей студента.

Типы связей: Один ко многим (1:N)

Один объект первой таблицы может быть связан со многими объектами второй таблицы, но дочерний объект связан ровно с одним родительским.

Пример в ЛШ «Квант»: Связь между студентами (students) и их достижениями (achievements).
  • Один студент может заработать много достижений.
  • Каждое конкретное достижение принадлежит строго одному студенту.
  • Правило: Внешний ключ (student_id) размещается в дочерней таблице на стороне «Многие» (то есть в achievements).

Типы связей: Многие ко многим (N:M)

Запись из таблицы А связана со множеством записей таблицы Б, и наоборот. Напрямую через одну ссылку связать их невозможно.

Пример: Студенты (students) и Мероприятия (events). Студент может записаться на много лекций, а на лекцию придет много ребят.

Решение: Создание связующей таблицы event_registrations:

CREATE TABLE event_registrations (
    student_id INTEGER REFERENCES students(user_id),
    event_id INTEGER REFERENCES events(id),
    PRIMARY KEY (student_id, event_id) -- составной ключ
);

Язык SQL и выборка данных (SELECT)

SQL — декларативный язык запросов. Мы пишем, что хотим получить, а не алгоритм поиска.

Выбрать все столбцы из таблицы пользователей:

SELECT * 
FROM users;

Выбрать только имя, фамилию и роль:

SELECT first_name, last_name, role 
FROM users;

Фильтрация данных: предложение WHERE

Используется для отсечения строк по условиям. Поддерживает логические связки AND, OR, NOT.

-- Выбрать студентов 1-го отряда с балансом не менее 150 коинов
SELECT user_id, balance 
FROM students 
WHERE cohort_number = 1 AND balance >= 150;
Операторы сравнения: = (равно), <> или != (не равно), >, <, >=, <=.

Поиск по шаблону (LIKE) и работа с NULL

LIKE и шаблоны: % (любое количество символов), _ (один любой символ).

-- Поиск пользователей, у которых фамилия начинается на "Ива"
SELECT first_name, last_name FROM users WHERE last_name LIKE 'Ива%';

Проверка на пустоту (NULL): Сравнение = NULL не работает. Нужно использовать IS NULL / IS NOT NULL.

-- Выбрать преподавателей, у которых не заполнена биография
SELECT user_id, subject FROM teachers WHERE bio IS NULL;

Сортировка (ORDER BY) и Ограничение (LIMIT)

Сортировка выполняется по возрастанию (ASC, по умолчанию) или убыванию (DESC).

-- Вывести топ-3 студентов по количеству заработанных квант-коинов
SELECT user_id, balance 
FROM students 
WHERE balance > 0
ORDER BY balance DESC 
LIMIT 3;
Сочетание ORDER BY и LIMIT часто применяется для формирования рейтингов, лидербордов и пагинации на страницах сайта.

Объединение таблиц: INNER JOIN vs LEFT JOIN

Поскольку данные нормализованы и лежат в разных таблицах, нам нужно склеивать их в одном запросе по ключевому полю.

  • INNER JOIN (Внутреннее объединение): Выводит строки только тогда, когда условие склеивания совпало в обеих таблицах.
  • LEFT JOIN (Левое внешнее объединение): Выводит все строки левой таблицы, а для полей правой таблицы подставляет NULL, если соответствий не найдено.
-- Пример INNER JOIN: Соединить профиль студента с его ФИО
SELECT u.first_name, u.last_name, s.cohort_number
FROM students s
JOIN users u ON s.user_id = u.id;

Агрегатные функции и GROUP BY

Агрегатные функции сжимают множество строк в одно значение: COUNT(), SUM(), AVG(), MIN(), MAX().

-- Посчитать сумму баллов по каждому отряду
SELECT cohort_number, SUM(balance) AS total_score, COUNT(*) AS students_count
FROM students
GROUP BY cohort_number;
Важнейшее правило: Любое поле в секции SELECT, не обернутое в агрегатную функцию, обязано присутствовать в блоке GROUP BY.

Фильтрация групп: HAVING против WHERE

  • WHERE — фильтрует строки до группировки. Не имеет доступа к агрегатам.
  • HAVING — фильтрует группы после агрегации и группировки.
-- Выбрать отряды, в которых средний балл студентов строго больше 100
SELECT cohort_number, AVG(balance) AS avg_bal
FROM students
GROUP BY cohort_number
HAVING AVG(balance) > 100;

Модификация данных: INSERT, UPDATE, DELETE

1. Добавление данных (INSERT)

INSERT INTO users 
(username, first_name, last_name, role) 
VALUES ('pupil_1', 'Олег', 'Петров', 'student');

2. Изменение данных (UPDATE)

UPDATE students 
SET balance = balance + 50 
WHERE cohort_number = 1;

3. Удаление данных (DELETE)

DELETE FROM users 
WHERE id = 4;
Осторожно! Забытый фильтр WHERE в запросах UPDATE и DELETE приведет к обновлению или удалению всей таблицы!

Логический порядок выполнения запроса СУБД

Хотя мы пишем ключевые слова в одном порядке, СУБД выполняет их совершенно в другом:

  1. FROM и JOIN (СУБД определяет, с какими таблицами работать)
  2. WHERE (Отбрасываются ненужные строки)
  3. GROUP BY (Строки делятся на группы)
  4. HAVING (Отбрасываются группы, не подходящие под условия)
  5. SELECT (Вычисляются результирующие колонки и выражения)
  6. ORDER BY (Результаты сортируются)
  7. LIMIT (Обрезается лишний вывод)
