Проблема: Хранение данных на сайте лагеря

Нашему сайту «Квант» нужно надежно хранить расписание, новости и сетку мероприятий. Почему обычные файлы (TXT или JSON) не подойдут?

  • Медленный поиск: Найти мероприятие на конкретную дату в файле на 500 МБ потребует чтения всего диска.
  • Конфликты записи: Если вожатый Вася добавляет новость, а вожатая Лена в ту же секунду меняет расписание, файл перезапишется с ошибкой.
  • Связи данных: Сложно связать новость с конкретным мероприятием.
Решение: СУБД SQLite

Мы используем реляционную базу данных SQLite. Она поставляется вместе с Django, хранит все данные в одном файле db.sqlite3 в корне проекта и идеально подходит для учебных и небольших проектов.

Что такое ORM?

Обычно для работы с реляционными БД программисты пишут запросы на языке SQL. Но в Django мы используем технологию ORM.

ORM (Object-Relational Mapping) — объектно-реляционное отображение. Технология, которая автоматически связывает таблицы базы данных с классами языка Python.

Благодаря ORM мы можем создавать таблицы, делать выборки и сохранять данные, вообще не зная синтаксиса SQL и работая только с объектами Python.

Соответствие между БД и Python

ORM сопоставляет концепции реляционной БД с концепциями Объектно-Ориентированного Программирования:

В Базе Данных (СУБД) В Django ORM (Python) Пример в проекте "Квант"
Таблица (Table) Класс Python (наследник models.Model) class News(models.Model)
Столбец / Колонка (Column) Поле модели (атрибут класса) title = models.CharField(...)
Строка в таблице (Row / Record) Экземпляр класса (объект модели) Конкретная новость о дискотеке

Преимущества использования ORM

  • Безопасность: Django автоматически экранирует все данные. Это полностью защищает сайт от хакерских атак типа SQL Injection.
  • Портативность (СУБД-независимость): Мы пишем код один раз. Если мы захотим перенести сайт лагеря с SQLite на мощный PostgreSQL, нам нужно будет изменить только 6 строчек в файле settings.py!
  • Скорость разработки: Написание Python-кода значительно быстрее составления громоздких SQL-запросов.
Сравнение запросов: 
# SQL Запрос:
SELECT * FROM news 
WHERE created_at = '2026-06-30';

# Django ORM:
News.objects.filter(
    created_at='2026-06-30'
)

Создание моделей в Django

Все модели нашего приложения описываются в файле models.py и наследуются от класса django.db.models.Model.

from django.db import models

class News(models.Model):
    # Поля модели (колонки таблицы) описываются как атрибуты класса:
    title = models.CharField(max_length=200)
    content = models.TextField()
    created_at = models.DateField(auto_now_add=True)

Наследуясь от models.Model, наш класс получает встроенную логику для сохранения в БД, удаления, поиска и валидации.

Поля моделей: CharField и TextField

CharField — Короткая строка
  • Используется для заголовков, имен, коротких текстов.
  • Обязательно требует параметр max_length (максимальное число символов). Это нужно СУБД для резервации памяти.
title = models.CharField(
    max_length=200
)
TextField — Длинный текст
  • Используется для подробного описания событий, текстов новостей, статей.
  • Не имеет ограничения по длине на уровне Django (БД сама выделяет динамический объем).
content = models.TextField()

Поля моделей: IntegerField и DateField

IntegerField — Целое число
  • Хранит только целые значения (как положительные, так и отрицательные).
  • Пример: номера кабинетов в расписании, количество участников.
classroom = models.IntegerField()
DateField — Работа с датами
  • Хранит дату (год, месяц, день) без указания точного времени.
  • Параметр auto_now_add=True автоматически заполняет поле текущей датой при создании записи.
date = models.DateField()

Поле ImageField

Для загрузки изображений (обложек мероприятий, фотографий к новостям) используется ImageField.

image = models.ImageField(upload_to='news_images/', blank=True, null=True)
Важные требования для ImageField:
  1. Для работы с изображениями в Python должна быть установлена библиотека Pillow (pip install Pillow).
  2. Параметр upload_to указывает подпапку внутри директории медиафайлов, куда Django физически сохранит файл.

Важные параметры полей

Параметр Описание Пример
default Задает дефолтное значение для поля, если оно не передано. default="Аудитория 1"
null=True Разрешает базе данных хранить пустое значение NULL. null=True (для обложек/изображений)
blank=True Разрешает не заполнять поле в веб-формах (например, в админке). blank=True (описание может быть пустым)
verbose_name Человекочитаемое имя поля для отображения в админ-панели. verbose_name="Время начала"

Модель Event (Событие лагеря)

Опишем модель для мероприятий ЛФМШ «Квант» (лекции, спортивные часы, вечерние дела):

from django.db import models

class Event(models.Model):
    title = models.CharField(max_length=200, verbose_name="Название события")
    description = models.TextField(verbose_name="Описание события")
    date = models.DateField(verbose_name="Дата проведения")
    start_time = models.CharField(max_length=5, verbose_name="Время начала") # Например, "17:00"
    cover = models.ImageField(upload_to='events/', blank=True, null=True, verbose_name="Обложка")

    def __str__(self):
        return f"{self.date} | {self.start_time} - {self.title}"

Модель News (Новость лагеря)

Опишем модель для новостной ленты летней школы:

from django.db import models

class News(models.Model):
    title = models.CharField(max_length=200, verbose_name="Заголовок")
    content = models.TextField(verbose_name="Текст новости")
    created_at = models.DateField(auto_now_add=True, verbose_name="Дата публикации")
    image = models.ImageField(upload_to='news/', blank=True, null=True, verbose_name="Изображение")

    def __str__(self):
        return f"{self.created_at} | {self.title}"

Модель Schedule (Расписание занятий)

Расписание учебных пар для физико-математических групп лагеря:

from django.db import models

class Schedule(models.Model):
    date = models.DateField(verbose_name="Дата занятий")
    time_slot = models.CharField(max_length=50, verbose_name="Временной интервал") # Например, "09:00 - 10:30"
    classroom = models.IntegerField(verbose_name="Номер кабинета")

    def __str__(self):
        return f"{self.date} | {self.time_slot} (Каб. {self.classroom})"

Класс Meta: Метаданные моделей

Для настройки поведения модели используется внутренний класс Meta:

class Schedule(models.Model):
    date = models.DateField(verbose_name="Дата")
    time_slot = models.CharField(max_length=50, verbose_name="Интервал")
    classroom = models.IntegerField(verbose_name="Кабинет")

    class Meta:
        verbose_name = "Пункт расписания"
        verbose_name_plural = "Расписание занятий"
        ordering = ['date', 'time_slot'] # Сортировка по дате, затем по времени

Параметр ordering задает сортировку по умолчанию. Знак минус (например, '-date') отсортирует записи в обратном порядке (по убыванию).

Что такое миграции и зачем они нужны?

Когда вы создаете класс в models.py, база данных о нем ничего не знает. Нам нужно как-то перенести эту структуру в СУБД.

Миграции — это файлы-инструкции на языке Python, которые описывают изменения структуры вашей базы данных. Они играют роль «системы контроля версий» для БД.

Жизненный цикл изменений:

  1. Вносим изменения в файл models.py (создаем класс, добавляем или удаляем поле).
  2. Создаем файл миграции (команда makemigrations).
  3. Применяем миграцию к базе данных (команда migrate).

Создание миграций: makemigrations

После написания моделей в models.py выполните команду в терминале:

python manage.py makemigrations

Django сравнивает текущие модели с предыдущими файлами миграций и автоматически генерирует новый файл-чертеж в папке migrations/ внутри приложения (например, 0001_initial.py).

Важно: На этом этапе база данных на диске еще не изменилась! Создан только файл-инструкция, описывающий, какие изменения нужно будет сделать.

Применение миграций: migrate

Чтобы физически создать таблицы или добавить колонки в СУБД, примените миграции:

python manage.py migrate

Django считывает все новые файлы миграций, переводит их инструкции в SQL-команды под используемую СУБД (в нашем случае SQLite) и выполняет их в базе данных.

Как Django понимает, какие миграции уже применены?
В базе данных создается таблица django_migrations. В нее записываются имена всех выполненных файлов миграций. Django повторно их не запускает.

База данных под капотом

  • Именование таблиц: По умолчанию Django создает таблицы с именем в формате имяПриложения_имяМодели в нижнем регистре (например, kvant_event).
  • Уникальный идентификатор: Django автоматически добавляет целочисленное автоинкрементное поле id в каждую модель. Это первичный ключ (Primary Key).
  • С каждым новым объектом его id автоматически увеличивается на 1: 1, 2, 3 и т.д.
Никогда не удаляйте файлы миграций вручную, если проект уже запущен! Это приведет к рассинхронизации кода моделей и структуры реальной БД.

Django Shell — Интерактивная консоль

Чтобы взаимодействовать с базой данных из терминала, обычный Python REPL не подойдет, так как он не знает о настройках проекта. Используйте команду:

python manage.py shell

Эта команда запускает консоль Python, предварительно импортируя настройки settings.py и инициализируя Django.

Перед работой с моделями в консоли их обязательно нужно импортировать:
from kvant.models import Event, News, Schedule

CRUD: Create (Создание данных)

Есть два способа создать запись в таблице через Django ORM:

Способ 1: Сначала в памяти, затем save() 
# Создаем объект в оперативной памяти
event = Event(
    title="Открытие смены Квант",
    description="Торжественная линейка",
    date="2026-06-30",
    start_time="10:00"
)
# Записываем в базу данных
event.save()
Способ 2: Метод objects.create() 
# Объект сразу записывается в БД
event = Event.objects.create(
    title="Олимпиадная математика",
    description="Лекция по теории чисел",
    date="2026-06-30",
    start_time="11:30"
)
# event.id уже заполнен автоматически!

CRUD: Read (Чтение и выборка данных)

Для получения записей используется менеджер objects, возвращающий коллекцию QuerySet:

  • Получить абсолютно все записи: 
    all_news = News.objects.all()
  • Фильтрация (выбрать записи по условию): 
    news_today = News.objects.filter(created_at="2026-06-30")
  • Исключение (выбрать все, кроме): 
    other_days_news = News.objects.exclude(created_at="2026-06-30")

CRUD: Read (Получение по ID и сортировка)

Поиск по уникальному ID: .get() 
# Возвращает один конкретный объект
news = News.objects.get(id=1)
Внимание! Метод .get() вызовет ошибку DoesNotExist, если запись не найдена, или MultipleObjectsReturned, если найдено несколько записей.
Сортировка: .order_by() 
# Сортировка по кабинетам (по возрастанию)
Schedule.objects.order_by('classroom')

# Сортировка по дате (по убыванию)
Schedule.objects.order_by('-date')

Field Lookups (Условия фильтрации)

Для детального поиска Django использует двойное подчеркивание __ после имени поля:

  • Поиск подстроки без учета регистра: __icontains 
    News.objects.filter(title__icontains="физика")
  • Сравнения: __gt (больше) и __lt (меньше) 
    # Кабинеты с номером больше 300
Schedule.objects.filter(classroom__gt=300)
  • Вхождение в список значений: __in 
    Schedule.objects.filter(classroom__in=[101, 102, 105])

CRUD: Update (Изменение данных)

Способ 1: Изменение конкретного объекта 
# Находим объект в БД
lesson = Schedule.objects.get(id=3)
# Меняем атрибут
lesson.classroom = 404
# Сохраняем изменения в базу
lesson.save()
Способ 2: Массовое обновление (на QuerySet) 
# Переносим все занятия 
# из 101 кабинета в 202
Schedule.objects.filter(
    classroom=101
).update(classroom=202)

Метод .update() выполняется сразу на уровне БД одним SQL-запросом.

CRUD: Delete (Удаление данных)

Удаление одного объекта: 
event = Event.objects.get(id=2)
# Полное удаление из базы данных
event.delete()
Массовое удаление (на QuerySet): 
# Удаляем все события на дату
Event.objects.filter(
    date="2026-06-25"
).delete()
Внимание: Вызов Event.objects.all().delete() мгновенно сотрет абсолютно все мероприятия из базы данных лагеря. Это действие необратимо!

Отображение объектов: Метод __str__

Без переопределения метода __str__ Django выведет объект в виде <Event: Event object (1)>. Это крайне неудобно при отладке.

class Event(models.Model):
    title = models.CharField(max_length=200)
    date = models.DateField()

    # Переопределяем метод строкового представления:
    def __str__(self):
        return f"{self.date} | {self.title}"
Теперь при вызове print(event) или при просмотре списка событий в админ-панели мы увидим понятное: 2026-06-30 | Открытие смены Квант.

Резюме: Алгоритм работы с моделями

  1. Описать модель как класс в models.py (наследование от models.Model).
  2. Добавить поля с нужными типами (CharField, TextField, DateField и др.).
  3. Определить метод __str__ и class Meta.
  4. Создать файлы миграций: python manage.py makemigrations.
  5. Применить изменения к базе: python manage.py migrate.
  6. Использовать Django Shell или View для CRUD-операций.
